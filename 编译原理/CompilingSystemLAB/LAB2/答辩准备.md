# 实验二答辩准备：语义分析

这份文档只说明实验二新增的语义分析部分。实验一已经完成的词法分析、语法分析和语法树构造不展开。

## 二、总体实现思路

语义分析没有直接写在 Bison 规则里，而是单独写成 `semantic.c`，对实验一生成的语法树做遍历。

整体流程：

1. 从语法树根结点 `Program` 开始。
2. 依次处理每个外部定义 `ExtDef`。
3. 遇到变量定义、函数定义、结构体定义时，把名字和类型加入符号表。
4. 遇到语句和表达式时，查询符号表并推导表达式类型。
5. 如果发现语义错误，按格式输出：

```txt
Error type X at Line Y: message.
```

入口函数：

```c
void semantic_analyze(TreeNode root)
```

它初始化三个表，然后遍历 `ExtDefList`，最后释放符号表。

## 三、数据结构设计

### 1. 类型系统 `Type`

`semantic.c` 中用 `Type` 表示 C-- 类型：

```c
struct Type {
    TypeKind kind;
    union {
        BasicKind basic;
        struct {
            Type *elem;
            int size;
        } array;
        struct {
            char *name;
            Field *fields;
        } structure;
    } u;
};
```

支持三类类型：

- `TYPE_BASIC`：基本类型，只有 `int` 和 `float`。
- `TYPE_ARRAY`：数组类型，保存元素类型和数组大小。
- `TYPE_STRUCT`：结构体类型，保存结构体名字和域链表。

函数没有作为普通表达式类型参与运算，而是单独放在函数符号表中。

### 2. 结构体域和函数参数 `Field`

```c
struct Field {
    char *name;
    Type *type;
    Field *next;
};
```

`Field` 是链表结点，用在两个地方：

- 结构体中的域列表。
- 函数形参列表。

### 3. 三张符号表

代码中维护三张全局链表：

```c
static VarSymbol *variables = NULL;
static FuncSymbol *functions = NULL;
static StructSymbol *structures = NULL;
```

分别表示：

- `variables`：变量表，包括全局变量、局部变量和函数形参。
- `functions`：函数表，记录函数名、返回类型、参数类型和参数个数。
- `structures`：结构体表，记录结构体名和域列表。

为什么变量表不做作用域？

因为本实验只做必做部分，指导书假设 4 是“所有变量作用域都是全局的，所有变量不能重名”。嵌套作用域是选做 2.2，所以这里故意不实现作用域栈。

## 十一、答辩时可以这样概括

本实验在实验一语法树的基础上增加了一个独立的语义分析模块。模块中设计了 `Type` 表示基本类型、数组和结构体，用 `Field` 链表表示结构体域和函数参数，并维护变量表、函数表和结构体表。语义分析从 `Program` 的 `ExtDefList` 开始遍历，定义类结点负责填符号表，表达式和语句类结点负责查符号表、推导类型并检查 1-17 类语义错误。数组类型比较忽略大小，只比较维数和基类型；结构体类型采用名等价。当前实现严格对应必做要求，没有实现函数声明、嵌套作用域和结构等价这些选做内容。

### 1. 从程序入口开始讲

```c

```

它外面还有一层判断，在 `lab2_code/main.c` 第 29 行：

```c

```

这表示只有当前面的词法分析和语法分析没有发现错误时，才进入语义分析。`root` 是实验一语法分析阶段构造出来的语法树根结点。实验二并不重新做词法和语法分析，而是在这个语法树基础上继续做语义检查。

接下来可以讲 `semantic_analyze`。它定义在 `lab2_code/semantic.c` 第 763 行：

```c
void semantic_analyze(TreeNode root)
```

这个函数是整个语义分析模块的总入口。它主要做三件事：

1. 初始化三张符号表：变量表、函数表、结构体表。
2. 从语法树根结点 `Program` 的第一个孩子 `ExtDefList` 开始遍历。
3. 对每个外部定义调用 `analyze_ext_def`，最后释放符号表。

可以这样讲：

> 语义分析从 `semantic_analyze(root)` 开始。这个函数不直接分析所有语法规则，而是从 `Program -> ExtDefList` 开始，逐个处理外部定义。外部定义可能是变量定义、结构体定义或者函数定义，所以后续的语义检查就围绕这些语法单元展开。

### 2. 讲核心数据结构

讲完入口后，不要马上讲递归函数，应该先解释语义分析依赖的数据结构。因为后面所有检查都围绕“类型”和“符号表”进行。

#### 2.1 Type 类型系统

`Type` 结构体在 `lab2_code/semantic.c` 第 65 行：

```c
struct Type {
```

这个结构表示 C-- 中变量和表达式的类型。它支持三类类型：

- `TYPE_BASIC`：基本类型，也就是 `int` 和 `float`。
- `TYPE_ARRAY`：数组类型，用递归方式保存数组元素类型。
- `TYPE_STRUCT`：结构体类型，保存结构体名字和结构体域链表。

可以这样讲：

> 我用 `Type` 统一表示语义分析中的类型。基本类型直接保存是 `int` 还是 `float`；数组类型保存当前维度大小和元素类型，所以多维数组可以递归表示；结构体类型保存结构体名和域列表。因为必做要求采用结构体名等价，所以结构体名在类型比较时很关键。

如果助教问数组怎么表示，可以举例：

```c
int a[10][2];
```

它在内部可以理解为：

```txt
ARRAY(size=10)
  ARRAY(size=2)
    BASIC(int)
```

#### 2.2 Field 域链表

`Field` 结构体在 `lab2_code/semantic.c` 第 92 行：

```c
struct Field {
```

它有两个用途：

1. 表示结构体域，例如 `struct Position { float x, y; };` 中的 `x` 和 `y`。
2. 表示函数参数，例如 `int func(int a, float b)` 中的 `a` 和 `b`。

可以这样讲：

> 结构体域和函数参数本质上都需要保存“名字 + 类型 + 下一个”，所以我统一用 `Field` 链表表示。这样结构体域检查和函数实参形参匹配都可以复用这套结构。

#### 2.3 三张符号表

三张符号表定义在 `lab2_code/semantic.c` 第 148 行开始：

```c
static VarSymbol *variables = NULL;
static FuncSymbol *functions = NULL;
static StructSymbol *structures = NULL;
```

它们分别表示：

- `variables`：变量表，保存普通变量、数组变量、结构体变量和函数形参。
- `functions`：函数表，保存函数名、返回类型、参数列表和参数个数。
- `structures`：结构体表，保存结构体名和结构体域。

可以这样讲：

> 我把变量、函数和结构体分成三张表保存，因为它们需要检查的内容不一样。变量表主要检查变量是否定义和变量重名，函数表主要检查函数调用和函数重复定义，结构体表主要检查结构体名是否定义以及结构体域访问。

这里还要强调一个点：

> 本实验只做必做部分，必做假设中规定所有变量包括函数形参都是全局作用域，所以我没有实现作用域栈。不同函数里的局部变量也不能重名，这是和选做 2.2 不同的地方。

### 3. 讲外部定义的遍历流程

数据结构讲完后，可以进入语法树遍历流程。

#### 3.1 analyze_ext_def

`analyze_ext_def` 定义在 `lab2_code/semantic.c` 第 798 行：

```c
static void analyze_ext_def(TreeNode node)
```

它处理外部定义 `ExtDef`。对应的文法主要有三种：

```txt
ExtDef -> Specifier ExtDecList SEMI
ExtDef -> Specifier SEMI
ExtDef -> Specifier FunDec CompSt
```

可以这样讲：

> `analyze_ext_def` 是外层遍历的核心函数。每遇到一个 `ExtDef`，我先调用 `analyze_specifier` 得到类型说明。然后看第二个孩子是什么：如果是 `ExtDecList`，说明是全局变量定义；如果是 `FunDec`，说明是函数定义；如果只是 `Specifier SEMI`，通常就是结构体定义，而结构体处理已经在 `analyze_specifier` 里面完成。

这里会涉及错误类型 4。`lab2_code/semantic.c` 第 811-812 行附近检查函数名是否重复：

```c
if (find_function(func_name) != NULL) {
    report_error(4, second->line, "Redefined function \"%s\"", func_name);
}
```

可以解释：

> 函数定义时，我会先查函数表，如果同名函数已经存在，就报错误类型 4。

#### 3.2 analyze_specifier

`analyze_specifier` 定义在 `lab2_code/semantic.c` 第 863 行：

```c
static Type *analyze_specifier(TreeNode node)
```

它把语法树中的类型说明转换成内部 `Type`。

可以这样讲：

> `Specifier` 可能是普通类型 `int`、`float`，也可能是结构体类型。对于 `int` 和 `float`，我直接创建基本类型；对于结构体，则交给 `analyze_struct_specifier` 继续处理。

#### 3.3 analyze_struct_specifier

`analyze_struct_specifier` 定义在 `lab2_code/semantic.c` 第 894 行：

```c
static Type *analyze_struct_specifier(TreeNode node)
```

它处理两类结构体语法：

```txt
StructSpecifier -> STRUCT OptTag LC DefList RC
StructSpecifier -> STRUCT Tag
```

可以这样讲：

> 如果是 `STRUCT OptTag LC DefList RC`，说明这是结构体定义，我会分析结构体内部的 `DefList`，收集所有域，形成 `Field` 链表。如果结构体有名字，就加入结构体表。  
> 如果是 `STRUCT Tag`，说明是在引用一个已经定义过的结构体，比如 `struct Position p;`，这时我会查结构体表。如果查不到，就报错误类型 17。

错误类型 17 的代码位置在 `lab2_code/semantic.c` 第 901-902 行：

```c
report_error(17, node->line, "Undefined structure \"%s\"", id->value.text);
```

错误类型 16 的检查在 `add_structure` 中，函数定义在 `lab2_code/semantic.c` 第 576 行，具体报错在第 583-584 行：

```c
report_error(16, line, "Duplicated name \"%s\"", name);
```

可以解释：

> 结构体定义时，如果结构体名和已有结构体或变量重复，就报错误类型 16。

### 7. 重点讲表达式分析 analyze_exp

最核心的函数是 `analyze_exp`，定义在 `lab2_code/semantic.c` 第 1236 行：

```c
static Type *analyze_exp(TreeNode node)
```

这部分答辩时一定要讲清楚，因为 17 类错误里很多都在表达式分析中检查。

可以这样开头：

> `analyze_exp` 的作用是分析一个表达式，并返回这个表达式的类型。它会根据 `Exp` 结点的子结点形状判断表达式种类，比如变量、常量、函数调用、赋值、算术运算、数组访问、结构体域访问等。分析子表达式时采用递归方式，底层表达式先返回类型，上层表达式再根据操作符做类型检查。

#### 7.1 变量使用：错误类型 1

单独的 `ID` 表示变量使用。检查位置在 `lab2_code/semantic.c` 第 1249-1250 行：

可以讲：

> 如果表达式是单独的 `ID`，我会在变量表里查找。如果找不到，说明变量使用前没有定义，报错误类型 1。

函数调用的检查在 `analyze_exp` 中处理 `ID LP ... RP`。

错误类型 11 的位置是 `lab2_code/semantic.c` 第 1331-1332 行：

```c

```

错误类型 2 的位置是第 1334-1335 行：

```c

```

错误类型 9 的一处位置是第 1344-1345 行：


实参和形参逐个比较在 `analyze_args` 中，函数定义在 `lab2_code/semantic.c` 第 1508 行附近，最终错误类型 9 的报错在第 1546-1547 行：


可以讲：

> 函数调用时先查函数表。如果函数表没有，但变量表里有同名变量，说明把变量当函数调用，报错误类型 11。如果函数表和变量表都没有，就报函数未定义，也就是错误类型 2。找到函数后，再通过 `analyze_args` 检查实参和形参的数量及类型，不匹配就报错误类型 9。

错误类型 12 的位置在第 1371-1372 行：

#### 7.6 运算表达式：错误类型 7

错误类型 7 在多个地方检查：

- 算术运算两边不是相同 `int` 或 `float`：第 1489-1490 行。

可以讲：

> 错误类型 7 是操作数类型不匹配。我分别在一元运算、逻辑运算、关系运算和算术运算中检查。算术运算只允许 int 或 float，并且两边类型要一致；逻辑运算和条件表达式要求 int；数组和结构体不能直接参与普通算术运算。

### 8. 讲类型比较 same_type

`same_type` 定义在 `lab2_code/semantic.c` 第 436 行：

```c
static int same_type(Type *left, Type *right)
```

可以这样讲：

> `same_type` 是统一的类型等价判断函数。基本类型必须完全相同；数组类型递归比较元素类型，不比较每一维大小；结构体类型采用名等价，即结构体名字相同才认为类型相同。

还可以补充：

> 如果某一边类型是 `NULL`，说明更早之前已经出现了未定义变量、未定义函数或未定义结构体这样的错误。为了避免连锁报错，我这里默认 `NULL` 不继续触发类型不匹配。
