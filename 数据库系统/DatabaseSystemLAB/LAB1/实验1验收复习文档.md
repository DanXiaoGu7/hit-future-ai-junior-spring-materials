## 一、实验目标

Lab1 的核心目标有两个：

1. 实现 LRU 页面替换策略。
2. 实现缓冲池管理器。

可以用一句话概括整个实验：

> 当数据库要访问某个页面时，先在缓冲池里找；找不到就从磁盘读入；缓冲池满了就用 LRU 找一个没人使用的页面淘汰；淘汰脏页前要写回磁盘。

## 二、基础概念

### 1. Page

`Page` 是数据库管理数据的基本单位。磁盘文件按页组织，缓冲池中也按页缓存数据。

在本项目中，一页大小是 `PAGE_SIZE = 4096` 字节。

### 2. Frame

`frame` 是缓冲池中的内存槽位。一个 frame 可以装一个 page。

可以这样理解：

```text
磁盘文件中有很多 page
缓冲池中有很多 frame
一个 frame 临时缓存一个 page
```

### 3. PageId

`PageId` 用来唯一标识一个页面，由两部分组成：

```cpp
int fd;
page_id_t page_no;
```

- `fd`：文件描述符，表示页面属于哪个打开的文件。
- `page_no`：页号，表示该页面是文件中的第几页。

只用 `page_no` 不够，因为不同文件都可能有第 0 页、第 1 页。所以必须用 `(fd, page_no)` 共同定位页面。

### 4. pin 与 unpin

`pin` 表示页面正在被使用，不能被淘汰。

`unpin` 表示调用者暂时用完该页面，页面的引用计数减少。

如果页面的 `pin_count_` 变成 0，说明没有人正在使用它，这时才可以加入 LRU 替换器，成为可淘汰页面。

### 5. dirty page

脏页是指内存中被修改过、但还没有写回磁盘的页面。

如果脏页被淘汰前不写回磁盘，修改就会丢失。

## 三、replacer 文件夹整体讲解

`src/replacer` 目录负责页面替换策略。替换策略只回答一个问题：

> 缓冲池满了，应该淘汰哪个 frame？

### 1. `replacer.h`

这是替换策略的抽象基类，定义所有替换器必须提供的接口。

核心代码如下：

```cpp
class Replacer {
   public:
    Replacer() = default;
    virtual ~Replacer() = default;

    virtual bool victim(frame_id_t *frame_id) = 0;

    virtual void pin(frame_id_t frame_id) = 0;

    virtual void unpin(frame_id_t frame_id) = 0;

    virtual size_t Size() = 0;
};
```

作用解释：

- `victim(frame_id_t *frame_id)`：选择一个可以淘汰的 frame，并通过参数返回。
- `pin(frame_id_t frame_id)`：表示该 frame 正在使用，不能被淘汰。
- `unpin(frame_id_t frame_id)`：表示该 frame 可以被淘汰。
- `Size()`：返回当前可淘汰 frame 的数量。

验收时可以说：

> `Replacer` 是替换策略接口，`LRUReplacer` 是它的具体实现。`BufferPoolManager` 不关心具体替换算法，只通过 `Replacer` 接口调用 `pin`、`unpin`、`victim`。

### 2. `lru_replacer.h`

这是 LRU 替换器的类声明。

关键成员如下：

```cpp
std::mutex latch_;
std::list<frame_id_t> LRUlist_;
std::unordered_map<frame_id_t, std::list<frame_id_t>::iterator> LRUhash_;
size_t max_size_;
```

作用解释：

- `latch_`：互斥锁，保证多线程访问时结构不乱。
- `LRUlist_`：保存当前可淘汰的 frame，并维护 LRU 顺序。
- `LRUhash_`：保存 `frame_id -> 链表节点位置`，方便 O(1) 删除指定 frame。
- `max_size_`：替换器最多管理多少 frame，通常等于缓冲池大小。

`LRUlist_` 和 `LRUhash_` 的关系：

```text
LRUlist_ 负责顺序
LRUhash_ 负责快速定位
两者保存的是同一批可淘汰 frame
插入和删除时必须同步更新
```

### 3. `lru_replacer.cpp`

这是我们实际补全 LRU 逻辑的文件。

LRU 规则：

```text
最久没有被使用的页面最先淘汰
```

在代码中：

- `LRUlist_` 头部是最旧的 frame。
- `LRUlist_` 尾部是最近 unpin 的 frame。
- `victim()` 从头部取。
- `unpin()` 插入尾部。
- `pin()` 从链表中删除。

## 四、我们写的 Replacer 代码段与解释

### 1. `LRUReplacer::victim`

完整代码：

```cpp
bool LRUReplacer::victim(frame_id_t* frame_id) {
    // victim 的目标：从“可淘汰集合”里选出最应该被淘汰的 frame。
    // LRU 的规则是：谁最久没有被使用，就淘汰谁。

    // LRUReplacer 可能被多个线程同时访问，所以每个公开操作都要先加锁。
    // 这里保护的是 LRUlist_ 和 LRUhash_ 这两个必须同步更新的数据结构。
    std::scoped_lock lock{latch_};

    // LRUlist_ 中只保存 pin_count 已经变成 0、可以被淘汰的 frame。
    // 如果链表为空，说明当前没有任何可淘汰页面，缓冲池管理器需要返回失败。
    if (frame_id == nullptr || LRUlist_.empty()) {
        return false;
    }

    // 本实现约定：
    // - 链表头部是最早进入 replacer 的 frame，即最久没有再次被 pin 的 frame。
    // - 链表尾部是最近一次 unpin 后加入 replacer 的 frame。
    // 因此 victim 时取链表头部，符合 LRU 的“淘汰最久未使用”策略。
    //
    // 举例：依次 unpin 1、2、3，则链表是 [1, 2, 3]。
    // 这时 1 最久没被使用，所以 victim 应该返回 1。
    *frame_id = LRUlist_.front();

    // LRUhash_ 记录 frame_id 到链表节点的映射。
    // 从链表删除 frame 时必须同步删除哈希表项，否则后续 pin/unpin 会看到过期位置。
    LRUhash_.erase(*frame_id);
    LRUlist_.pop_front();
    return true;
}
```

实现逻辑：

1. 先加锁，防止多个线程同时修改 LRU 链表和哈希表。
2. 如果 `frame_id` 为空或 `LRUlist_` 为空，说明没有可淘汰页面，返回 `false`。
3. 取 `LRUlist_.front()`，也就是最久未使用的 frame。
4. 同时从 `LRUhash_` 和 `LRUlist_` 删除该 frame。
5. 返回 `true`，并通过 `*frame_id` 告诉缓冲池淘汰哪个 frame。

验收回答：

> `victim` 的作用是在缓冲池满时选择一个可淘汰 frame。因为链表头部代表最久未使用的 frame，所以取 `front()`。删除时必须同步删除哈希表，否则会留下无效迭代器。

### 2. `LRUReplacer::pin`

完整代码：

```cpp
void LRUReplacer::pin(frame_id_t frame_id) {
    // pin 的目标：把一个 frame 从“可淘汰集合”里移除。
    // 原因：页面一旦被 fetch 或 new_page 返回给上层，就正在被使用，不能淘汰。
    std::scoped_lock lock{latch_};

    // pin 表示页面正在被使用，不能被替换算法选中。
    // 如果该 frame 不在 LRUhash_ 中，说明它本来就不在可淘汰集合里，直接返回即可。
    auto it = LRUhash_.find(frame_id);
    if (it == LRUhash_.end()) {
        return;
    }

    // 如果该 frame 在可淘汰集合中，需要同时从链表和哈希表删除。
    // 通过哈希表保存的迭代器可以 O(1) 定位链表节点。
    LRUlist_.erase(it->second);
    LRUhash_.erase(it);
}
```

实现逻辑：

1. `pin` 表示页面正在被使用，不能被淘汰。
2. 通过 `LRUhash_` 查找 frame 是否在 LRU 中。
3. 如果不在，说明它本来就不可淘汰，直接返回。
4. 如果在，用哈希表保存的迭代器从链表删除，再删除哈希表记录。

验收回答：

> `pin` 是把 frame 从可淘汰集合移除。因为 LRUhash_ 能直接定位链表节点，所以删除是 O(1)，不用遍历链表。

### 3. `LRUReplacer::unpin`

完整代码：

```cpp
void LRUReplacer::unpin(frame_id_t frame_id) {
    // unpin 的目标：把一个 frame 加入“可淘汰集合”。
    // 但只有当 BufferPoolManager 中该页的 pin_count_ 已经变成 0 时才会调用到这里。
    std::scoped_lock lock{latch_};

    // unpin 只有在页面 pin_count 变为 0 时才会由 BufferPoolManager 调用。
    // 此时 frame 才可以重新进入可淘汰集合。
    //
    // 这里过滤两类情况：
    // 1. frame_id 越界，防止错误输入破坏替换器状态。
    // 2. frame 已经在 LRUhash_ 中，说明之前已经加入过，重复 unpin 不能重复插入。
    if (frame_id < 0 || static_cast<size_t>(frame_id) >= max_size_ || LRUhash_.count(frame_id) != 0) {
        return;
    }

    // 新 unpin 的 frame 放到链表尾部，表示它是“最近变为可淘汰”的页面。
    // 这样后续 victim 会优先淘汰链表头部更旧的页面。
    //
    // 举例：当前链表 [1, 2]，现在 unpin(3)，链表变为 [1, 2, 3]。
    // 下一次 victim 仍然先淘汰 1，而不是刚刚 unpin 的 3。
    LRUlist_.push_back(frame_id);
    auto it = LRUlist_.end();
    --it;
    LRUhash_[frame_id] = it;
}
```

实现逻辑：

1. `unpin` 表示页面已经没人使用，可以加入可淘汰集合。
2. 检查 `frame_id` 是否越界，避免非法编号破坏数据结构。
3. 检查 `LRUhash_` 中是否已经存在该 frame，避免重复插入。
4. 把 frame 插入 `LRUlist_` 尾部，代表它最近才变为可淘汰。
5. 用 `end()` 得到尾后迭代器，再 `--it` 指向刚插入的节点。
6. 在 `LRUhash_` 中保存 `frame_id -> it`，方便以后 O(1) 删除。

验收回答：

> `unpin` 是把页面加入可淘汰集合。它插入链表尾部，因为刚 unpin 的页面是最近变成可淘汰的，不应该马上被淘汰。哈希表保存它在链表中的位置，方便 `pin` 时快速删除。

### 4. `LRUReplacer::Size`

完整代码：

```cpp
size_t LRUReplacer::Size() {
    // Size 在并发测试中也可能和 pin/unpin/victim 同时执行，
    // 因此读取链表大小时同样需要加锁。
    std::scoped_lock lock{latch_};
    return LRUlist_.size();
}
```

实现逻辑：

1. 加锁读取链表大小。
2. 返回当前可淘汰 frame 数量。

验收回答：

> `Size` 返回当前 LRU 中可被淘汰的 frame 数量。加锁是为了避免并发时读到正在修改中的链表状态。

## 五、storage 文件夹整体讲解

`src/storage` 目录负责存储层。Lab1 最关键的是 `Page` 和 `BufferPoolManager`。

### 1. `page.h`

`page.h` 定义了页面标识 `PageId` 和页面对象 `Page`。

#### `PageId`

关键代码：

```cpp
struct PageId {
    int fd;
    page_id_t page_no = INVALID_PAGE_ID;

    friend bool operator==(const PageId &x, const PageId &y) { return x.fd == y.fd && x.page_no == y.page_no; }
    bool operator<(const PageId& x) const {
        if(fd < x.fd) return true;
        return page_no < x.page_no;
    }

    std::string toString() {
        return "{fd: " + std::to_string(fd) + " page_no: " + std::to_string(page_no) + "}";
    }

    inline int64_t Get() const {
        return (static_cast<int64_t>(fd << 16) | page_no);
    }
};
```

解释：

- `fd` 表示页面属于哪个文件。
- `page_no` 表示页面在该文件中的页号。
- `operator==` 让两个 `PageId` 可以比较是否相同。
- `PageIdHash` 和 `std::hash<PageId>` 让 `PageId` 可以作为 `unordered_map` 的 key。

验收回答：

> `PageId` 唯一定位一个页面，因为页面必须同时知道属于哪个文件和是文件中的第几页。

#### `Page`

关键代码：

```cpp
class Page {
    friend class BufferPoolManager;

   public:
    Page() { reset_memory(); }

    ~Page() = default;

    PageId get_page_id() const { return id_; }

    inline char *get_data() { return data_; }

    bool is_dirty() const { return is_dirty_; }

   private:
    void reset_memory() { memset(data_, OFFSET_PAGE_START, PAGE_SIZE); }

    PageId id_;

    char data_[PAGE_SIZE] = {};

    bool is_dirty_ = false;

    int pin_count_ = 0;
};
```

解释：

- `id_`：该页的唯一编号。
- `data_`：实际存储页面数据的内存数组，大小为 `PAGE_SIZE`。
- `is_dirty_`：是否被修改过。
- `pin_count_`：当前有多少调用者正在使用该页。
- `friend class BufferPoolManager`：允许 `BufferPoolManager` 直接访问和修改 `Page` 的私有成员。

验收回答：

> `Page` 是缓冲池中缓存磁盘页的对象，里面既有实际数据 `data_`，也有用于缓冲池管理的元数据，比如 `id_`、`is_dirty_` 和 `pin_count_`。

### 2. `buffer_pool_manager.h`

这是缓冲池管理器的声明文件。

核心成员：

```cpp
size_t pool_size_;
Page *pages_;
std::unordered_map<PageId, frame_id_t, PageIdHash> page_table_;
std::list<frame_id_t> free_list_;
DiskManager *disk_manager_;
Replacer *replacer_;
std::mutex latch_;
```

解释：

- `pool_size_`：缓冲池中 frame 的数量。

- `page_table_`：从 `PageId` 找到对应 `frame_id`。

析构函数释放 `pages_` 和 `replacer_`。

验收回答：

> `BufferPoolManager` 是缓冲池的核心类，负责管理内存页、页表、空闲 frame、替换策略和磁盘读写。

### 3. `disk_manager.h/.cpp`

解释：

## 六、我们写的 BufferPoolManager 代码段与解释

### 1. `find_victim_page`

完整代码：

```cpp
bool BufferPoolManager::find_victim_page(frame_id_t* frame_id) {
    // 这个辅助函数只回答一个问题：
    // “现在有没有一个 frame 可以拿来放页面？”
    //
    // 它不负责读磁盘，也不负责写磁盘，只返回可用 frame 的编号。

    // 缓冲池找可用 frame 有两个来源：
    // 1. free_list_：从未使用或已经 delete_page 释放的空闲 frame。
    // 2. replacer_：缓冲池满时，由 LRU 选择 pin_count 为 0 的可淘汰 frame。
    //
    // 必须优先使用 free_list_，因为空闲 frame 不需要淘汰旧页，也不需要处理写回。
    if (!free_list_.empty()) {
        *frame_id = free_list_.front();
        free_list_.pop_front();
        return true;
    }

    // free_list_ 为空说明所有 frame 都装过页面。
    // 此时只能从 replacer_ 中找可淘汰页；如果所有页都被 pin，victim 会返回 false。
    return replacer_->victim(frame_id);
}
```

实现逻辑：

1. 优先从 `free_list_` 取空闲 frame。
2. 如果没有空闲 frame，再调用 LRU 的 `victim()`。
3. 如果 LRU 找不到可淘汰 frame，返回 `false`。

验收回答：

> 找可用 frame 时先用空闲 frame，因为不用淘汰旧页；没有空闲 frame 时才用 LRU 找 `pin_count_ == 0` 的 frame。

### 2. `fetch_page`

完整代码：

```cpp
Page* BufferPoolManager::fetch_page(PageId page_id) {
    // fetch_page 的作用：获取一个已经存在的磁盘页。
    //
    // 它有两种可能：
    // 1. 页面已经在缓冲池：直接返回内存页。
    // 2. 页面不在缓冲池：找 frame，从磁盘 read_page 读进来。

    // BufferPoolManager 的页表、空闲链表、页面元数据都是共享状态，
    // 所以对外接口统一加锁，保证并发测试下不会出现状态竞争。
    std::scoped_lock lock{latch_};

    // INVALID_PAGE_ID 不是一个真实磁盘页，不能 fetch。
    if (page_id.page_no == INVALID_PAGE_ID) {
        return nullptr;
    }

    // 情况 1：目标页已经在缓冲池中。
    // page_table_ 保存 PageId -> frame_id 的映射，可以 O(1) 定位页面所在 frame。
    //
    // 这也是 page_table_ 的核心作用：避免每次都遍历整个 pages_ 数组查找。
    auto table_it = page_table_.find(page_id);
    if (table_it != page_table_.end()) {
        Page* page = &pages_[table_it->second];

        // fetch 表示上层开始使用该页面，所以 pin_count_ 加 1。
        // pin_count_ > 0 的页面不能被淘汰。
        //
        // 例如同一个页面被两个地方同时 fetch，pin_count_ 就可能是 2。
        // 只有两个地方都 unpin 后，它才能重新进入 LRU。
        page->pin_count_++;

        // 如果这个页面之前 pin_count 为 0，它可能在 replacer_ 里。
        // 现在重新被使用，需要从可淘汰集合中移除。
        replacer_->pin(table_it->second);
        return page;
    }

    // 情况 2：目标页不在缓冲池中，需要找一个 frame 装入磁盘页。
    // 如果没有空闲 frame，且 LRU 也找不到可淘汰 frame，说明所有页面都被 pin，fetch 失败。
    frame_id_t frame_id = INVALID_FRAME_ID;
    if (!find_victim_page(&frame_id)) {
        return nullptr;
    }

    Page* page = &pages_[frame_id];

    // 如果选中的 frame 原来装着一个有效页面，需要先处理旧页。
    // 注意：从 free_list_ 取出的全新 frame 的 page_no 通常是 INVALID_PAGE_ID。
    if (page->id_.page_no != INVALID_PAGE_ID) {
        // 旧页是脏页时，说明内存内容比磁盘新。
        // 淘汰前必须写回磁盘，否则之前的修改会丢失。
        if (page->is_dirty_) {
            disk_manager_->write_page(page->id_.fd, page->id_.page_no, page->data_, PAGE_SIZE);
        }

        // 旧页即将离开缓冲池，页表映射必须删除。
        // 否则后续 fetch 旧 PageId 会错误地定位到这个 frame。
        //
        // 举例：frame 5 原来存 page A，现在要改存 page B。
        // 如果不删除 page A 的映射，page_table_ 会误以为 page A 还在 frame 5。
        page_table_.erase(page->id_);
    }

    // 清空 frame 中的旧数据，然后从磁盘读入目标页。
    page->reset_memory();
    disk_manager_->read_page(page_id.fd, page_id.page_no, page->data_, PAGE_SIZE);

    // 更新页面元数据和页表。
    // 新读入的页面正在被 fetch 调用者使用，所以 pin_count_ 初始化为 1。
    // 刚从磁盘读入，与磁盘一致，所以 is_dirty_ 为 false。
    page->id_ = page_id;
    page->pin_count_ = 1;
    page->is_dirty_ = false;
    page_table_[page_id] = frame_id;

    // 防御性调用 pin，确保该 frame 不在 replacer_ 中。
    replacer_->pin(frame_id);
    return page;
}
```

实现逻辑：

1. 加锁，保护缓冲池共享状态。
2. 如果页面已经在缓冲池中：
   - 找到对应 frame。
   - `pin_count_++`。
   - 从 LRU 中移除。
   - 返回页面。
3. 如果页面不在缓冲池中：
   - 找可用 frame。
   - 如果旧页是脏页，先写回磁盘。
   - 删除旧页页表映射。
   - 从磁盘读入目标页。
   - 更新 `id_`、`pin_count_`、`is_dirty_` 和 `page_table_`。

验收回答：

> `fetch_page` 先查页表。如果页面在缓冲池中，就增加引用计数并返回；如果不在，就找 frame，必要时淘汰旧页，脏页先写回，再从磁盘读入目标页。

### 3. `unpin_page`

完整代码：

```cpp
bool BufferPoolManager::unpin_page(PageId page_id, bool is_dirty) {
    // unpin_page 的作用：告诉缓冲池“我暂时用完这个页面了”。
    //
    // 注意：unpin 不是删除页面，也不是写回页面。
    // 它只是把 pin_count_ 减 1；只有减到 0 时，页面才可以被 LRU 淘汰。
    std::scoped_lock lock{latch_};

    // 只有在缓冲池中的页面才能 unpin。
    auto table_it = page_table_.find(page_id);
    if (table_it == page_table_.end()) {
        return false;
    }

    Page* page = &pages_[table_it->second];

    // pin_count_ 已经为 0 表示没有使用者，继续 unpin 是非法操作。
    if (page->pin_count_ <= 0) {
        return false;
    }

    // is_dirty 是本次调用者告诉缓冲池“我是否修改了页面”。
    // 只能在 true 时置脏，不能在 false 时清脏：
    // 因为页面可能之前已经被其他调用修改过，false 只表示本次没有新修改。
    //
    // 这是验收常问点：
    // is_dirty=false 不能写成 page->is_dirty_=false，否则会把之前的修改标记清掉。
    if (is_dirty) {
        page->is_dirty_ = true;
    }

    // 每次 unpin 只释放一次使用引用。
    page->pin_count_--;

    // 只有 pin_count_ 变成 0，页面才真正没人使用，才能交给 LRU 管理。
    if (page->pin_count_ == 0) {
        replacer_->unpin(table_it->second);
    }
    return true;
}
```

实现逻辑：

1. 查页表，不存在则失败。
2. 如果 `pin_count_ <= 0`，说明没有可释放引用，返回失败。
3. 如果本次调用修改了页面，则置脏。
4. `pin_count_--`。
5. 如果引用计数变成 0，加入 LRU。

验收回答：

> `unpin_page` 不是删除页，只是减少页面使用计数。只有计数减到 0，页面才可以被 LRU 淘汰。`is_dirty=false` 不能清除脏标记，因为页面可能之前已经被修改过。

### 4. `flush_page`

完整代码：

```cpp
bool BufferPoolManager::flush_page(PageId page_id) {
    // flush_page 的作用：把缓冲池中某一页的内存内容写回磁盘。
    // 它不改变 pin_count_，也不把页面从缓冲池中删除。
    std::scoped_lock lock{latch_};

    // flush_page 只能刷新当前在缓冲池中的有效页面。
    auto table_it = page_table_.find(page_id);
    if (table_it == page_table_.end() || page_id.page_no == INVALID_PAGE_ID) {
        return false;
    }

    Page* page = &pages_[table_it->second];

    // 实验要求 flush 是强制刷盘：
    // 不管页面是否为脏页，不管 pin_count_ 是否大于 0，都写回磁盘。
    disk_manager_->write_page(page->id_.fd, page->id_.page_no, page->data_, PAGE_SIZE);

    // 写回后内存和磁盘一致，可以清除脏页标记。
    page->is_dirty_ = false;
    return true;
}
```

实现逻辑：

1. 查页表，页面不在缓冲池中则失败。
2. 调用 `write_page` 强制写回磁盘。
3. 写回后清除脏标记。

验收回答：

> `flush_page` 是强制刷盘，不管页面是否为脏页，也不管是否正在被 pin，只要页面在缓冲池中就写回。

### 5. `new_page`

完整代码：

```cpp
Page* BufferPoolManager::new_page(PageId* page_id) {
    // new_page 的作用：创建一个新的磁盘页，并把这个新页放进缓冲池。
    //
    // 它和 fetch_page 的区别：
    // - fetch_page 是读取“已经存在的 page_no”。
    // - new_page 是先 allocate_page 分配一个新的 page_no。
    std::scoped_lock lock{latch_};

    // 新建页面同样需要一个 frame。
    // 如果所有 frame 都被 pin，无法淘汰，new_page 失败。
    frame_id_t frame_id = INVALID_FRAME_ID;
    if (!find_victim_page(&frame_id)) {
        return nullptr;
    }

    Page* page = &pages_[frame_id];

    // 如果该 frame 原来保存了有效旧页，需要先处理旧页再复用 frame。
    if (page->id_.page_no != INVALID_PAGE_ID) {
        // 脏页淘汰前必须写回，保证数据持久化。
        if (page->is_dirty_) {
            disk_manager_->write_page(page->id_.fd, page->id_.page_no, page->data_, PAGE_SIZE);
        }

        // 删除旧页的页表映射，避免旧 PageId 继续指向这个已复用的 frame。
        page_table_.erase(page->id_);
    }

    // DiskManager 按文件 fd 自增分配 page_no。
    // 传入的 page_id 已经带有 fd，这里补全新页面编号。
    //
    // PageId 由 fd 和 page_no 组成：
    // - fd 表示这个页属于哪个打开的文件。
    // - page_no 表示它是该文件里的第几页。
    page_id->page_no = disk_manager_->allocate_page(page_id->fd);

    // 新页开始时是空白页，先重置内存，再设置元数据。
    page->reset_memory();
    page->id_ = *page_id;

    // new_page 返回的页面马上交给调用者写入，所以 pin_count_ 为 1。
    // 此时页面内容刚初始化为空白，和下面写入磁盘后的内容一致，所以不是脏页。
    page->pin_count_ = 1;
    page->is_dirty_ = false;

    // 建立新 PageId 到 frame 的映射。
    page_table_[*page_id] = frame_id;

    // 新页正在被调用者使用，不能被 LRU 淘汰。
    replacer_->pin(frame_id);

    // 立即把空白页写入磁盘。
    // 这样即使后续该页未被修改就被 fetch，也能从磁盘读到完整 PAGE_SIZE 数据。
    disk_manager_->write_page(page_id->fd, page_id->page_no, page->data_, PAGE_SIZE);
    return page;
}
```

实现逻辑：

1. 找可用 frame。
2. 如果旧页有效且是脏页，先写回。
3. 删除旧页页表映射。
4. 调用 `allocate_page` 分配新页号。
5. 清空页面内容，初始化元数据。
6. 建立新页表映射。
7. 把新页从 LRU 中移除，因为它正在被使用。
8. 写入空白页到磁盘。

验收回答：

> `new_page` 是创建新页，不是读取已有页。它先分配新页号，再初始化页面，设置 `pin_count_ = 1`，加入页表。写入空白页是为了保证磁盘上已经存在完整页面，后续读取不会短读。

### 6. `delete_page`

完整代码：

```cpp
bool BufferPoolManager::delete_page(PageId page_id) {
    // delete_page 的作用：释放缓冲池中某个页面占用的 frame。
    //
    // 这里的“删除”主要是清理缓冲池状态，并把 frame 放回 free_list_。
    // DiskManager::deallocate_page 在当前实验中是空实现，所以磁盘页回收不是重点。
    std::scoped_lock lock{latch_};

    // 如果页面不在缓冲池中，缓冲池内没有状态需要清理。
    // DiskManager::deallocate_page 当前是空实现，但保留调用符合接口语义。
    auto table_it = page_table_.find(page_id);
    if (table_it == page_table_.end()) {
        disk_manager_->deallocate_page(page_id.page_no);
        return true;
    }

    frame_id_t frame_id = table_it->second;
    Page* page = &pages_[frame_id];

    // 仍被使用的页面不能删除，否则调用者手中的 Page* 会失效。
    if (page->pin_count_ != 0) {
        return false;
    }

    // 删除前写回页面，保证已修改的数据不会因为释放 frame 而丢失。
    disk_manager_->write_page(page->id_.fd, page->id_.page_no, page->data_, PAGE_SIZE);
    disk_manager_->deallocate_page(page_id.page_no);

    // 页面从缓冲池移除，页表和 replacer 都要同步清理。
    page_table_.erase(table_it);
    replacer_->pin(frame_id);

    // 重置 frame，使它重新变成空闲 frame。
    page->reset_memory();
    page->id_.fd = INVALID_PAGE_ID;
    page->id_.page_no = INVALID_PAGE_ID;
    page->pin_count_ = 0;
    page->is_dirty_ = false;

    // 释放后的 frame 放回 free_list_，后续优先复用。
    free_list_.push_back(frame_id);
    return true;
}
```

实现逻辑：

1. 如果页面不在缓冲池中，直接返回成功。
2. 如果页面在缓冲池中但 `pin_count_ != 0`，说明仍被使用，不能删除。
3. 可以删除时先写回磁盘。
4. 删除页表映射。
5. 从 LRU 中移除。
6. 重置页面元数据。
7. 把 frame 放回 `free_list_`。

验收回答：

> `delete_page` 只有在页面 `pin_count_ == 0` 时才能删除。删除后要清理页表、清理 LRU 状态、重置页面，并把 frame 放回空闲链表。

### 7. `flush_all_pages`

完整代码：

```cpp
void BufferPoolManager::flush_all_pages(int fd) {
    // flush_all_pages 的作用：把某个文件在缓冲池中的所有页面都写回磁盘。
    // 参数 fd 用来筛选“属于哪个文件”的页面。
    std::scoped_lock lock{latch_};

    // 遍历页表，找出所有属于指定文件 fd 的页面。
    // 这里不调用 flush_page，是因为当前已经持有 latch_，
    // flush_page 内部也会加同一把锁，直接调用会造成死锁。
    for (const auto& entry : page_table_) {
        const PageId& page_id = entry.first;
        if (page_id.fd != fd || page_id.page_no == INVALID_PAGE_ID) {
            continue;
        }

        // 实验要求 flush_all_pages 将该文件在缓冲池中的所有页面写回磁盘。
        Page* page = &pages_[entry.second];
        disk_manager_->write_page(page->id_.fd, page->id_.page_no, page->data_, PAGE_SIZE);
        page->is_dirty_ = false;
    }
}
```

实现逻辑：

1. 加锁。
2. 遍历 `page_table_`。
3. 只处理指定 `fd` 的页面。
4. 写回磁盘。
5. 清除脏标记。

为什么不调用 `flush_page()`：

`flush_all_pages()` 已经持有 `latch_`，而 `flush_page()` 里面也会加 `latch_`。如果直接调用，可能同一线程重复申请同一把非递归锁，导致死锁。

验收回答：

> `flush_all_pages` 遍历页表，把属于指定文件的页面全部写回。不能直接调用 `flush_page`，因为两个函数都会加同一把锁，容易死锁。

## 七、Lab1 阅读代码问题标准回答

### 1. `LRUlist_` 的作用是什么？

`LRUlist_` 保存当前可以被淘汰的 frame，也就是对应页面 `pin_count_` 已经变成 0 的 frame。

链表头部是最久未使用的 frame，优先被淘汰；链表尾部是最近一次 `unpin` 后加入的 frame。

### 2. `LRUhash_` 的作用是什么？

`LRUhash_` 保存 `frame_id -> LRUlist_ 中节点位置` 的映射。

它的作用是让 `pin(frame_id)` 可以 O(1) 找到并删除链表中的指定 frame，而不用遍历整个链表。

### 3. `LRUlist_` 和 `LRUhash_` 的关系是什么？

它们保存同一批可淘汰 frame。

- `LRUlist_` 维护淘汰顺序。
- `LRUhash_` 负责快速定位。

插入和删除时必须同步更新，否则会出现链表和哈希表不一致。

### 4. `Page::is_dirty_` 的作用是什么？

`is_dirty_` 表示页面是否被修改过。

如果为 `true`，说明内存中的页面比磁盘新。该页面被淘汰、删除或刷盘时必须写回磁盘，否则修改会丢失。

### 5. `Page::pin_count_` 的作用是什么？

`pin_count_` 表示当前有多少调用者正在使用页面。

只要 `pin_count_ > 0`，页面就不能被淘汰。只有当 `pin_count_` 降到 0，页面才能加入 LRU 替换器。

### 6. `BufferPoolManager::page_table_` 的作用是什么？

`page_table_` 是 `PageId -> frame_id` 的哈希表。

它用于判断页面是否已经在缓冲池中，并快速找到该页面位于哪个 frame。

### 7. `BufferPoolManager::free_list_` 的作用是什么？

`free_list_` 保存当前空闲 frame 的编号。

缓冲池未满时优先从 `free_list_` 取 frame；只有没有空闲 frame 时，才调用 LRU 替换策略淘汰旧页面。

## 八、验收高频追问

### 1. 为什么需要加锁？

因为 `LRUlist_`、`LRUhash_`、`page_table_`、`free_list_`、`pin_count_` 都是共享状态，可能被多个线程同时访问。

不加锁可能导致：

- 同一个 frame 被重复插入 LRU。
- 链表删了但哈希表没删。
- 页表指向已经被复用的 frame。
- 页面还在使用却被淘汰。
- 并发测试失败甚至程序崩溃。

一句话回答：

> 加锁是为了让“检查状态和修改状态”成为一个不可被其他线程打断的整体。

### 2. 为什么 `is_dirty=false` 不能清除脏标记？

因为 `is_dirty=false` 只表示“本次 unpin 的调用者没有修改页面”，不代表页面之前没有被别人修改过。

如果写成：

```cpp
page->is_dirty_ = is_dirty;
```

那么之前已经置脏的页面可能被错误清成干净页，淘汰时就不会写回磁盘，导致数据丢失。

### 3. `fetch_page` 和 `new_page` 的区别是什么？

`fetch_page` 获取已经存在的页面：

```text
已有 PageId -> 查缓冲池 -> 不在就从磁盘读
```

`new_page` 创建新的页面：

```text
先 allocate_page 分配新 page_no -> 初始化空白页 -> 加入缓冲池
```

### 4. 为什么淘汰旧页前要删除 page_table_ 中旧映射？

因为一个 frame 被复用后会装入新页面。

如果不删除旧映射，`page_table_` 会误以为旧页面还在这个 frame 中。后续再 fetch 旧页面时，可能拿到新页面的数据。

### 5. 为什么脏页淘汰前必须写回？

脏页表示内存内容已经被修改，磁盘内容还没更新。

如果直接淘汰，内存中的修改会消失，磁盘仍是旧数据。

### 6. 为什么 `delete_page` 要检查 `pin_count_`？

如果 `pin_count_ != 0`，说明页面还被调用者使用。

此时删除会导致调用者手里的 `Page*` 指向被重置或复用的 frame，造成错误。

### 7. 为什么 `flush_all_pages` 不调用 `flush_page`？

因为两个函数内部都会加 `latch_`。

如果 `flush_all_pages` 持有锁时再调用 `flush_page`，同一线程会再次申请同一把非递归锁，可能死锁。

## 九、整体流程示例

### 示例 1：读取一个不在缓冲池中的页面

```text
fetch_page(page_id)
  -> page_table_ 查不到
  -> find_victim_page 找 frame
  -> 如果旧页是脏页，写回磁盘
  -> 删除旧页 page_table_ 映射
  -> DiskManager::read_page 读入目标页
  -> 设置 pin_count_ = 1
  -> 建立新 page_table_ 映射
  -> 返回 Page*
```

### 示例 2：用完一个页面

```text
unpin_page(page_id, is_dirty)
  -> 查 page_table_
  -> 如果 is_dirty 为 true，设置脏页标记
  -> pin_count_ 减 1
  -> 如果 pin_count_ 变成 0，加入 LRU
```

### 示例 3：缓冲池满了创建新页

```text
new_page(page_id)
  -> find_victim_page
  -> free_list_ 为空，调用 LRU victim
  -> 如果 victim 是脏页，写回磁盘
  -> 删除旧映射
  -> allocate_page 分配新页号
  -> 初始化空白页
  -> 建立新映射
  -> 返回新页
```
